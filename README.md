# Camera Switching Automation (Presenter vs Audience) with UI Button Control

This macro was created to improve the camera experience in Cisco Room systems that use separate cameras for the audience and the presenter. In many meeting rooms, the audience camera should remain the default view so remote participants can see the room naturally, but when a presenter steps into the PresenterTrack area, the meeting should automatically focus on the presenter without requiring manual camera control.

The macro solves this by keeping the audience camera as the main video source by default while PresenterTrack runs in the background. PresenterTrack is used only as a detection mechanism until a presenter is found. When the codec reports that a presenter has been detected, the macro automatically switches the main video source to the configured presenter camera. When the presenter is no longer detected, the macro waits for a short configurable delay before switching back to the audience camera. This delay helps avoid unnecessary switching if the presenter briefly moves out of the tracking area.

The purpose of the macro is to create a more natural and automated meeting room experience. It reduces the need for users, presenters, or support staff to manually select cameras during a call. The room can stay focused on the audience when no one is presenting, but automatically highlight the presenter when presentation activity begins. This is especially useful in training rooms, town halls, classrooms, briefing rooms, and hybrid meeting spaces where the correct camera view depends on whether someone is actively presenting.

The macro includes a RoomOS UI panel that allows users or technicians to control the behavior directly from the touch interface. One toggle enables or disables automatic camera switching, while another toggle controls whether monitoring should only run during active calls. This makes the solution flexible: it can be used as a fully automatic call-based camera behavior, or as a manually enabled background monitoring feature depending on the room requirements.

To make the switching behavior stable, the macro includes rate limiting and delayed fallback logic. Rate limiting prevents the codec from switching cameras too frequently, while the presenter-loss delay prevents camera flapping when PresenterTrack briefly loses detection. The macro can also automatically recover PresenterTrack if its status becomes Off, helping keep background presenter detection active during operation.

The macro also detects whether Microsoft Teams Rooms is installed on the device. This does not change the camera switching logic, but it automatically adjusts where the UI panel is placed. On standard RoomOS systems, the panel is placed on the home screen. When MTR is detected, the panel is moved to the control panel, making the UI placement more appropriate for that operating mode.

Overall, this macro was created to provide reliable automatic camera switching between audience and presenter views while keeping the user experience simple. It combines Cisco PresenterTrack detection, configurable camera source selection, RoomOS UI controls, call-state awareness, and safety mechanisms to deliver a hands-off camera workflow for rooms where presenter visibility is important but the audience view should remain the default.
